Módulo 1 · Fundamentos
01

Por que TypeScript?

TypeScript é JavaScript com uma camada extra: tipos. Você já sabe escrever JS — TypeScript adiciona a possibilidade de dizer explicitamente "essa variável é um número", "essa função recebe uma string e devolve um boolean", etc.

O grande ganho é que o editor (e o compilador) passam a avisar erros antes de rodar o código, em vez de você descobrir o problema só em produção.

JS vs TS: todo código JavaScript válido já é, tecnicamente, código TypeScript válido (com tipos implícitos). TypeScript não substitui o JS — ele é compilado para JS antes de rodar no navegador ou no Node. Ninguém "roda" .ts diretamente; sempre existe uma etapa de compilação.
exemplo.ts
Experimente: descomente a linha com saudacao(42) e rode. O compilador vai recusar, porque 42 é número e a função espera uma string. Isso é TypeScript "pegando" o erro antes mesmo de executar.

tipos-basicos.ts
Inferência de tipo: na maior parte do tempo você não precisa escrever : string em toda variável — o TypeScript "infere" (deduz) o tipo pelo valor inicial. Escrever o tipo é mais útil em parâmetros de função e quando a variável começa sem valor definido.

03

Arrays e tuplas

Pra tipar um array, você escreve o tipo dos elementos seguido de []. Já uma tupla é um array de tamanho fixo onde cada posição tem um tipo específico — útil quando a ordem e a quantidade de itens importam.

arrays-tuplas.ts
Pensa assim: um array tipado é uma caixa que só aceita um tipo de item, em qualquer quantidade. Uma tupla é mais como uma ficha de cadastro com campos fixos: posição 0 sempre é nome, posição 1 sempre é idade — a estrutura é rígida.

Módulo 2 · Estruturas de tipos
05

Interfaces

Uma interface descreve o "formato" que um objeto deve ter: quais chaves existem e qual o tipo de cada uma. É como um contrato — qualquer objeto que "assinar" essa interface precisa seguir exatamente essa estrutura.

interfaces.ts
Repare no JS gerado: a interface inteira desaparece na compilação. Ela existe só pra checagem de tipos — não gera nenhum código real em tempo de execução, diferente de uma classe.
06

Type aliases

type permite dar um nome pra qualquer tipo — não só objetos, mas também uniões, tuplas, funções, o que for. É parecido com interface, mas mais flexível.

type-aliases.ts
InterfaceType
Só descreve formato de objetosDescreve qualquer tipo (união, tupla, primitivo...)
Pode ser "reaberta" e estendida depoisNão pode ser reaberta
Convenção comum pra objetos/classes públicasConvenção comum pra unions e tipos utilitários
Na prática: pra objetos simples, tanto faz usar interface quanto type. A maioria dos times escolhe uma convenção e segue — o importante é entender que type é obrigatório quando você precisa de union types, como no exemplo acima.
07

Union e intersection types

Union (|) significa "esse tipo pode ser A ou B". Intersection (&) significa "esse tipo precisa ser A e B ao mesmo tempo" — combina dois tipos em um só.

union-intersection.ts
Pensa assim: union é um cardápio "escolha A ou B". Intersection é um combo "você leva A e B juntos, obrigatoriamente".
08

Literal types

Além de tipos genéricos como string, o TypeScript permite restringir um valor a opções exatas — por exemplo, só aceitar as strings "pequeno", "médio" ou "grande".

literal-types.ts
Por que isso é útil: em vez de usar uma string qualquer (que aceita literalmente qualquer texto, incluindo erros de digitação), literal types garantem que só valores válidos e previstos passem pelo compilador.
09

Enums

enum é outra forma de representar um conjunto fixo de valores nomeados — parecido com literal types, mas vira uma estrutura de verdade no JavaScript gerado (diferente de interface/type).

enums.ts
Diferente de interface/type: clique em "ver JS gerado" e note que o enum vira um objeto de verdade no JavaScript final. Por isso enums têm um pequeno custo em tempo de execução — muitos projetos modernos preferem literal types ("pendente" | "enviado") justamente por não gerarem nenhum código extra.
Módulo 3 · Tipos especiais
10

any, unknown e never

Três tipos especiais que representam situações diferentes de "eu não sei o tipo exato":

  • any — desliga a checagem de tipos completamente pra essa variável. Use o mínimo possível.
  • unknown — também aceita qualquer valor, mas te obriga a checar o tipo antes de usar. É o "any seguro".
  • never — representa um valor que nunca vai existir, geralmente usado em funções que sempre lançam erro ou nunca terminam.
any-unknown-never.ts
Regra prática: se você não sabe o tipo de algo (dados vindos de uma API, por exemplo), prefira sempre unknown a any. O unknown te obriga a validar antes de usar, evitando bugs silenciosos.

12

Generics

Generics permitem escrever uma função (ou classe) que funciona com qualquer tipo, mas mantendo a segurança de tipos — em vez de usar any e perder toda checagem. A letra T é convenção (poderia ser qualquer nome) e representa "um tipo a ser definido depois".

Pensa assim: uma função genérica é como uma forma de bolo universal — funciona pra qualquer massa (número, string, objeto), mas garante que o que sai é do mesmo tipo do que entrou.
generics.ts
Clique em "ver JS gerado": generics, assim como interfaces, somem completamente na compilação. Eles só existem pra garantir consistência de tipos durante o desenvolvimento.
generics-interface.ts
13

Utility types

TypeScript vem com tipos prontos que transformam outros tipos automaticamente. Os mais usados no dia a dia:

Utility typeO que faz
Partial<T>Torna todos os campos de T opcionais
Required<T>Torna todos os campos de T obrigatórios
Readonly<T>Torna todos os campos de T somente-leitura
Pick<T, K>Cria um tipo só com as chaves K escolhidas de T
Omit<T, K>Cria um tipo com todas as chaves de T, exceto K
utility-types.ts
Por que isso importa: em vez de duplicar uma interface inteira só pra remover ou tornar opcional um campo, utility types deixam você derivar um novo tipo a partir de um já existente. Isso evita repetição e mantém tudo sincronizado — se Usuario mudar, os tipos derivados acompanham.
14

Type narrowing (estreitamento de tipo)

Quando uma variável pode ser de mais de um tipo (union), o TypeScript consegue "estreitar" o tipo dentro de um bloco if, com base em checagens que você já usa em JS puro, como typeof e instanceof.

narrowing.ts
Isso é o que torna union types práticos: você declara que algo "pode ser A ou B", e dentro de cada checagem o compilador te dá acesso só aos métodos/propriedades válidos daquele tipo específico.
15

Decorators

Decorators são funções especiais que "decoram" (modificam ou observam) classes e seus membros, usando a sintaxe @nomeDoDecorator. São muito usados em frameworks como Angular e NestJS. É um recurso mais avançado — bom conhecer, mas menos comum no dia a dia comparado ao resto deste notebook.

Antes de rodar: decorators precisam de uma configuração especial do compilador (experimentalDecorators). O editor abaixo já está configurado pra isso funcionar.
decorators.ts
Vale notar: decorators de classe/método/propriedade (como acima) são um recurso mais antigo e ainda em evolução na linguagem. Você vai encontrá-los principalmente em frameworks específicos — não é algo que se usa em todo projeto TypeScript.
